# Feature Specification: 异动检测规则引擎

**Feature Branch**: `003-anomaly-detection`

**Created**: 2026-06-08

**Status**: Draft

**Input**: User description: "异动检测规则引擎"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 涨跌幅超阈值告警 (Priority: P1)

作为市场监控者，当某个 A 股指数涨跌幅超过预设阈值（如涨超 3% 或跌超 2%）时，我希望系统自动识别并生成异动事件，以便第一时间发现市场剧烈波动。

**Why this priority**: 涨跌幅是最直观的市场异动指标，直接反映市场情绪突变。

**Independent Test**: 请求大盘总览，若某指数涨跌幅超过阈值，返回结果中包含该指数的异动标记。

**Acceptance Scenarios**:

1. **Given** 上证指数当日涨幅为 3.5%，上限阈值为 3%，**When** 系统检测指数行情，**Then** 该指数被标记为"上涨异动"，事件包含指数名称、涨幅、触发时间。
2. **Given** 创业板指数当日跌幅为 2.8%，下限阈值为 2%，**When** 系统检测指数行情，**Then** 该指数被标记为"下跌异动"。
3. **Given** 所有指数涨跌幅均在阈值范围内，**When** 系统检测指数行情，**Then** 无异动事件产生。

---

### User Story 2 - 成交量突增检测 (Priority: P1)

作为市场监控者，当某个指数成交量较近期均值突增（如超过 2 倍），我希望系统能识别出来，因为成交量异动往往预示趋势转折。

**Why this priority**: 量价配合分析是技术分析的基础，成交量突变是重要的先行指标。

**Independent Test**: 某指数当日成交量超过近 5 日均值的 2 倍，验证系统生成异动事件。

**Acceptance Scenarios**:

1. **Given** 上证指数当日成交量为 1000 万手，近 5 日均值为 400 万手（2.5 倍），**When** 系统检测指数行情，**Then** 该指数被标记为"放量异动"。
2. **Given** 所有指数成交量均在正常范围（不超过 2 倍均值），**When** 系统检测指数行情，**Then** 无成交量异动事件。
3. **Given** 某指数历史数据不足 5 天，**When** 系统检测成交量，**Then** 跳过该指数的成交量异动检测（不报错）。

---

### User Story 3 - 连续涨跌检测 (Priority: P2)

作为市场监控者，当某个指数连续 N 个交易日上涨或下跌时，我希望系统能识别出趋势持续性，因为连续行情往往意味着趋势确立。

**Why this priority**: 连续涨跌是趋势强度的直接体现，补充单日涨跌幅检测的不足。

**Independent Test**: 某指数连续 5 日上涨，验证系统生成"连涨异动"事件。

**Acceptance Scenarios**:

1. **Given** 上证指数连续 5 个交易日上涨，规则定义为连涨 5 日触发，**When** 系统检测指数行情，**Then** 该指数被标记为"连涨异动"，含连续天数和累计涨幅。
2. **Given** 创业板指数连续 3 个交易日下跌，规则定义为连跌 3 日触发，**When** 系统检测指数行情，**Then** 该指数被标记为"连跌异动"。
3. **Given** 某指数连续天数未达阈值，**When** 系统检测，**Then** 不触发连续涨跌异动。

---

### User Story 4 - 异动事件查询 (Priority: P2)

作为 API 使用者，我希望通过专用接口查询当前或历史的异动事件列表，而非从行情数据中自行筛选，以便快速浏览市场异常情况。

**Why this priority**: 独立的异动查询接口提升可用性，便于下游系统集成。

**Independent Test**: 请求异动事件接口，验证返回当前检测到的所有异动事件。

**Acceptance Scenarios**:

1. **Given** 市场存在 2 个异动事件（1 个涨超阈值 + 1 个放量），**When** 用户请求异动事件列表，**Then** 返回 2 条事件记录，每条包含类型、指数、数值、阈值、时间。
2. **Given** 当前无异动事件，**When** 用户请求异动事件列表，**Then** 返回空列表，状态正常。

---

### Edge Cases

- 同一指数同时触发多项规则（涨超阈值 + 放量），生成多条独立事件还是合并？
- 规则阈值配置为负数或极端值（如涨跌幅阈值 100%）时如何处理？
- 成交量均值的计算基准日数（5 日）不足时（新股/新指数）的降级策略？
- 连续涨跌检测在遇到节假日断档时是否仍视为连续？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持基于涨跌幅阈值的异动检测：当日涨跌幅超过配置上限或下限时生成异动事件。
- **FR-002**: 系统 MUST 支持基于成交量突增的异动检测：当日成交量超过近 N 日均值的 M 倍时生成异动事件。
- **FR-003**: 系统 MUST 支持基于连续涨跌的异动检测：指数连续 N 个交易日同向变动时生成异动事件。
- **FR-004**: 系统 MUST 提供异动事件查询接口，返回当前检测到的所有异动事件列表。
- **FR-005**: 系统 MUST 支持通过配置定义检测规则（阈值、参数），无需修改代码。
- **FR-006**: 每个异动事件 MUST 包含：事件类型、指数标识、触发值、阈值/参数、触发时间。
- **FR-007**: 同一指数同时触发多条规则时，每条规则独立生成事件。

### Key Entities

- **检测规则 (DetectionRule)**: 规则类型（涨跌幅/成交量/连续涨跌）、参数（阈值、天数等）、启用状态。
- **异动事件 (AnomalyEvent)**: 事件类型、指数代码、指数名称、触发值、对比基准、触发时间、规则描述。
- **成交量基准 (VolumeBaseline)**: 某指数近 N 日成交量均值，用于成交量突增判断。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 涨跌幅检测准确率 100%（指数行情数据与公开数据一致，阈值判断为精确数学比较）。
- **SC-002**: 异动事件查询接口响应时间不超过当前大盘总览接口响应时间。
- **SC-003**: 规则配置修改后重启服务即可生效，无需重新编译。
- **SC-004**: 连续涨跌检测在 365 天历史数据范围内准确率 100%。

## Assumptions

- 成交量均值计算基准为近 5 个交易日，倍数阈值为 2 倍，可通过配置调整。
- 连续涨跌默认为连续 5 个交易日同向，可通过配置调整。
- 涨跌幅阈值默认为上限 3%、下限 -2%，可通过配置调整。
- 异动检测在每次请求大盘总览或指数行情时同步执行，不单独定时运行。
- 历史行情数据通过 Tushare API 实时获取（连续涨跌检测需要），不依赖本地数据库。
