# Feature Specification: 定时采集

**Feature Branch**: `004-scheduled-collection`

**Created**: 2026-06-09

**Status**: Draft

**Input**: User description: "定时采集"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 交易时段自动采集行情 (Priority: P1)

作为系统运维者，我希望系统在 A 股交易时段自动定时从 Tushare 拉取指数行情数据并缓存，而非等到用户请求时才实时查询，以保证数据随时可用且减少 API 重复调用。

**Why this priority**: 定时采集是数据底座的核心能力，支撑后续的异动监控、历史分析等功能，也是从"按需查询"升级为"持续监控"的第一步。

**Independent Test**: 启动服务后等待一个采集周期，检查内存或缓存中是否存在未被请求过的行情数据。

**Acceptance Scenarios**:

1. **Given** 当前为 A 股交易日且处于交易时段（9:30-15:00），**When** 系统启动并经过一个采集周期，**Then** 系统自动拉取所有追踪指数的行情数据并缓存。
2. **Given** 当前为非交易日（周末/节假日），**When** 系统运行中，**Then** 采集器不执行数据拉取（跳过当前周期）。
3. **Given** Tushare API 在某次采集时不可用，**When** 采集周期到达，**Then** 系统记录错误日志，跳过本次采集，不影响后续周期的执行。

---

### User Story 2 - 采集间隔可配置 (Priority: P1)

作为运维者，我希望定时采集的间隔（如每 5 分钟或每 10 分钟）可以通过配置调整，无需修改代码。

**Why this priority**: 不同使用场景对数据时效性要求不同，配置化是最基本的灵活性要求。

**Independent Test**: 修改配置文件中的采集间隔为 1 分钟，重启后观察两次采集的时间差是否约为 1 分钟。

**Acceptance Scenarios**:

1. **Given** 配置采集间隔为 5 分钟，**When** 系统在交易时段运行，**Then** 每 5 分钟执行一次数据采集。
2. **Given** 配置采集间隔为 0 或负数，**When** 系统启动，**Then** 采集器不启动（等同于禁用）。

---

### User Story 3 - 历史行情数据查询 (Priority: P2)

作为 API 使用者，我希望通过接口查询过去某个时间段的指数行情数据，用于分析趋势和回测，而非只能查看当前实时数据。

**Why this priority**: 历史数据是异动检测中成交量均值和连续涨跌计算的基础，也是未来分析功能的必要输入。

**Independent Test**: 采集运行一段时间后，请求历史行情接口指定日期范围，验证返回对应时间段的数据。

**Acceptance Scenarios**:

1. **Given** 系统已采集了最近 3 天的行情数据，**When** 用户请求最近 2 天的指数行情，**Then** 返回对应数据，按日期降序排列。
2. **Given** 请求的日期范围内无采集数据，**When** 用户请求历史行情，**Then** 返回空列表和正常状态码。
3. **Given** 部分指数在请求日期范围内数据不完整，**When** 用户请求历史行情，**Then** 返回可获取的数据，缺失部分注明。

---

### Edge Cases

- 采集器在执行过程中服务收到退出信号，如何处理（优雅关闭 vs 强制中断）？
- 连续多次采集失败（如 Tushare 长时间宕机）是否需要告警或降级？
- 历史数据缓存无限增长导致内存溢出，如何限制？
- 采集周期跨越交易时段边界（如 11:28 采集一次，下次是 11:33 但已午休），如何处理？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在交易时段自动按配置间隔周期性拉取所有追踪指数的行情数据。
- **FR-002**: 系统 MUST 在非交易时段（收盘后、周末、节假日）自动暂停采集。
- **FR-003**: 系统 MUST 支持通过配置调整采集间隔（分钟级），修改后重启生效。
- **FR-004**: 系统 MUST 将采集到的行情数据缓存到内存中，并支持历史查询。
- **FR-005**: 系统 MUST 在采集失败时记录错误日志，不中断采集循环。
- **FR-006**: 系统 MUST 提供服务优雅关闭时等待当前采集完成的能力。
- **FR-007**: 系统 MUST 提供历史行情查询接口，支持按日期范围筛选。

### Key Entities

- **采集调度器 (Collector)**: 管理采集循环、间隔控制、启停逻辑。
- **历史行情记录 (HistoricalQuote)**: 与 IndexQuote 结构相同，附加采集时间戳，按日期 + 指数代码索引。
- **采集配置**: 采集间隔（分钟）、最大历史记录条数、是否启用。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 交易时段内，连续两次采集的时间间隔与配置值的偏差不超过 30 秒。
- **SC-002**: 非交易时段采集数为 0（零次误触发）。
- **SC-003**: 历史行情查询接口响应时间不超过 100 毫秒（纯内存查询）。
- **SC-004**: 连续 10 次采集失败后系统仍正常运行（不崩溃、不阻塞 API）。

## Assumptions

- 默认采集间隔为 5 分钟，可在配置中调整。
- 内存中最多保留每个指数最近 30 个交易日的行情数据（约 6 周），超出自动淘汰。
- 采集使用与当前行情查询相同的 6 个 A 股指数（上证、深证、创业板、科创50、沪深300、中证500）。
- 交易日历判断复用已有的 `TradingCalendar` 模块。
- Phase 2 暂不引入数据库，历史数据存于内存结构。
- 采集器在服务启动时自动开始，无需手动触发。
