- Q: 收盘总结是否提供独立的 JSON 查询接口（GET），还是仅通过企业微信推送交付？ → A: 提供 GET 查询接口，返回当日收盘总结 JSON — 立即返回触发确认，后台推送，通过推送历史接口查询结果

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 收盘后自动收到当日总结 (Priority: P1)

作为投资者，我希望每个交易日下午收盘后，企业微信群能自动收到一条当日的市场总结消息，这样我无需打开任何系统就能快速了解今天大盘的整体表现、哪些指数涨了跌了、是否有异动。

**Why this priority**: 这是"主动触达"的核心价值。用户从"主动查询"变为"被动接收"，是系统从工具升级为助手的关键一步。

**Independent Test**: 在一个交易日下午 15:00 后等待 15 分钟，企业微信群收到一条 Markdown 总结消息，内容包含当日涨跌统计和异动汇总。

**Acceptance Scenarios**:

1. **Given** 今天是交易日，系统已采集当日行情数据，**When** 到达收盘推送时间（15:15），**Then** 企业微信群收到一条 Markdown 格式的收盘总结，包含各指数涨跌幅、异动事件数量、推送时间。
2. **Given** 今天是交易日但当日无任何异动事件，**When** 到达收盘推送时间，**Then** 企业微信群收到总结消息，异动部分显示"今日无异常"。
3. **Given** 今天是非交易日（周末/节假日），**When** 到达约定的推送时间，**Then** 系统不发送推送，不产生错误。
4. **Given** Webhook 推送未启用或未配置 Webhook URL，**When** 到达收盘推送时间，**Then** 系统跳过推送，记录日志，不影响其他功能运行。

---

### User Story 2 - 总结内容清晰完整 (Priority: P2)

作为投资者，我希望收盘总结的内容足够丰富，让我一眼就能掌握当日市场的关键信息：哪些指数表现最好/最差、整体涨跌比、是否有需要关注的异动。

**Why this priority**: 内容质量决定用户是否愿意持续关注推送。内容太少没价值，太多反而造成骚扰。

**Independent Test**: 查看历史推送记录中的收盘总结，验证包含涨跌统计、异动数量和交易日期。

**Acceptance Scenarios**:

1. **Given** 当日 6 个指数中 4 涨 2 跌，**When** 收盘总结生成，**Then** 消息包含每个指数的名称和涨跌幅（如"上证指数 +0.39%、深证成指 -0.19%"），以及"上涨 4 只，下跌 2 只"的汇总统计。
2. **Given** 当日出现 3 条异动事件，**When** 收盘总结生成，**Then** 消息异动部分列出异动数量、涉及的指数和类型（但不超过 5 条摘要，避免消息过长）。
3. **Given** 当日某指数涨幅超过 3%，**When** 收盘总结生成，**Then** 该指数被突出展示为"今日领涨"。
4. **Given** 当日某指数跌幅超过 2%，**When** 收盘总结生成，**Then** 该指数被标记为"今日领跌"。

---

### User Story 3 - 推送记录可追溯 (Priority: P3)

作为运维人员，我希望每天的收盘推送记录能被记录下来，方便事后确认推送是否成功、内容是否正确。

**Why this priority**: 追溯能力是运营兜底，优先级低于核心推送功能。

**Independent Test**: 推送发生后，调用推送历史接口验证收盘总结记录存在，状态为 success。

**Acceptance Scenarios**:

1. **Given** 收盘总结推送成功发送到企业微信，**When** 查询推送历史和收盘总结查询接口，**Then** 记录中包含该条总结（状态 success），查询接口返回与推送内容一致的 JSON 数据。
2. **Given** 收盘总结推送失败（如网络不可达），**When** 查询推送历史，**Then** 记录中该条状态为 failed，包含错误信息。

---

### Edge Cases

- 交易日但当日行情数据采集异常（如 Tushare 无响应）时，总结如何处理？
- 配置的推送时间在非交易时段是否仍然触发？
- 如果某天忘记配置 Webhook 但开启了总结功能，系统行为如何？
- 总结消息过长（如追踪指数数量较多且每条都有异动）时如何截断？
- 时区问题：A 股交易时间基于东八区，在非东八区环境中运行时推送时间是否正确？
- 服务在推送时间未运行、后续重启的情况如何处理？（已澄清：不补发，提供手动触发端点）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在每个交易日下午收盘后（默认 15:15）自动生成并推送一条收盘总结到企业微信。
- **FR-002**: 收盘总结 MUST 包含当日行情概览：每个追踪指数的名称和涨跌幅（逐个列出）、上涨/下跌/持平数量统计、领涨指数（最大涨幅）和领跌指数（最大跌幅）。
- **FR-003**: 收盘总结 MUST 包含当日异动汇总：异动事件总数，当数量 >0 时列出每条异动的指数和类型（最多 5 条，超出显示"等 N 条"）。
- **FR-004**: 系统 MUST 通过交易日历判断当日是否为交易日，非交易日不触发推送。
- **FR-005**: 收盘总结 MUST 复用现有的 Webhook 推送通道（notify 领域），推送记录写入推送历史缓冲区。
- **FR-006**: 推送失败（网络错误、Webhook 返回错误）MUST NOT 影响系统的其他功能，失败详情写入日志。
- **FR-007**: 推送时间 MUST 可配置（默认 15:15），格式为 HH:MM。
- **FR-008**: 收盘总结功能 MUST 可通过配置独立开关控制启用/禁用，与 Webhook 告警推送的开关解耦。
- **FR-009**: 当日行情数据获取失败时（如 Tushare 不可达），系统 MUST 尽力发送可用数据并在推送消息中明确标注"数据异常：部分数据获取失败"，而非静默失败或跳过推送。
- **FR-010**: 系统 MUST 提供一个手动触发端点，允许用户在交易日随时请求生成并推送当日收盘总结（异步触发，立即返回确认，后台完成推送，结果通过推送历史查询）。
- **FR-011**: 系统 MUST 提供 GET 查询接口，返回当日（或指定日期）收盘总结的 JSON 数据，包含涨跌统计、各指数涨跌幅、领涨/领跌指数、异动事件摘要，独立于企业微信推送。

### Key Entities

- **收盘总结配置 (DailySummaryConfig)**: 启用开关、推送时间（HH:MM）、追踪指数列表（复用现有配置）。
- **收盘总结内容 (DailySummary)**: 交易日期、涨跌统计（涨/跌/平数量）、领涨/领跌指数及其涨跌幅、异动事件摘要列表、数据获取状态。
- **收盘总结推送记录**: 复用现有 `DeliveryRecord`，通过特殊标记或事件类型区分"收盘总结"和"异动告警"推送。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 每个交易日下午 15:15 ± 1 分钟内，用户在企业微信群收到收盘总结消息。
- **SC-002**: 收盘总结消息（推送和 API 查询）包含至少 4 类信息：各指数涨跌幅、涨跌统计、领涨/领跌指数、异动事件数量。
- **SC-003**: 收盘总结推送成功率不低于 95%（在 Webhook 服务正常的前提下）。
- **SC-004**: 从收盘时间（15:00）到推送完成的端到端延迟不超过 3 分钟。
- **SC-005**: 用户可在 5 分钟内完成收盘总结功能的配置（修改 YAML + 重启服务）。

## Assumptions

- 收盘总结通过企业微信 Markdown 消息推送，复用现有 `WebhookSender`（企业微信群机器人）。
- 推送时间默认 15:15（收盘后 15 分钟），通过配置 `daily_summary.push_time` 调整。
- 收盘总结的数据来源为当日采集器采集的行情数据（`HistoryStore` 内存数据）。
- 交易日判断依赖现有的交易日历（`domain/market` 的 `TradingCalendar`）。
- 总结中异动摘要最多列出 5 条，超出部分以"等 N 条"折叠，避免消息体过大。
- 收盘总结推送也受 Webhook 推送重试策略保护（默认 3 次重试）。
- 功能默认关闭，需显式配置 `daily_summary.enabled: true` 启用。
- 领涨/领跌按涨跌幅绝对值排序，第一名为领涨/领跌指数。
