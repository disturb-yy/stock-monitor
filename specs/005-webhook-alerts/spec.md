# Feature Specification: Webhook 告警推送

**Feature Branch**: `005-webhook-alerts`

**Created**: 2026-06-09

**Status**: Draft

**Input**: User description: "异动检测触发后通过 Webhook 推送告警，支持企业微信"

## Clarifications

### Session 2026-06-09

- Q: 告警消息格式与内容？ → A: 企业微信 Markdown 格式，内容固定包含：指数名称、异动类型（中文）、触发值、阈值、触发时间。
- Q: 推送失败重试策略？ → A: 仅 5xx、429、网络超时重试；4xx（400/401/404）立即标记失败并记录日志。
- Q: 同一轮检测产生多条异动时的推送策略？ → A: 合并为一条消息，用分隔线区分各条异动。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 异动事件自动推送到企业微信 (Priority: P1)

作为运维人员，当系统检测到股票指数异动事件时，我希望自动将告警消息推送到企业微信群机器人，这样我无需时刻盯着系统界面也能第一时间收到市场异常通知。

**Why this priority**: 打通企业微信意味着告警从"系统内可见"升级为"主动触达"，是告警推送最核心的价值。

**Independent Test**: 配置企业微信 Webhook 地址后，模拟触发一次涨跌幅异动，验证企微群内收到包含异动详情（指数名称、异动类型、触发值、时间）的消息。

**Acceptance Scenarios**:

1. **Given** 已配置企业微信机器人 Webhook URL 且启用推送，**When** 异动检测引擎生成一个上涨异动事件（上证指数涨超 3%），**Then** 企业微信群收到一条 Markdown 消息，包含指数名称、异动类型、触发值和触发时间。
2. **Given** 已配置企业微信 Webhook URL 但网络不可达，**When** 异动事件产生，**Then** 系统记录推送失败日志，不影响异动检测和 API 的正常运行。
3. **Given** 未配置企业微信 Webhook URL，**When** 异动事件产生，**Then** 系统不尝试推送，不产生错误。
4. **Given** 同一轮检测产生多条异动事件（如 2 个指数同时触发），**When** 系统推送告警，**Then** 合并为一条 Markdown 消息，用分隔线区分各条异动。

---

### User Story 2 - 告警去重与冷却 (Priority: P2)

作为运维人员，当同一指数在短时间内反复触发同类型异动（如价格在阈值附近震荡），我不希望收到大量重复告警造成骚扰，系统应具备去重冷却机制。

**Why this priority**: 无冷却的重复告警会导致"狼来了"效应，降低运维人员对告警的信任度，是告警系统可用性的关键保障。

**Independent Test**: 连续 3 次产生同一指数 + 同一类型的异动事件（间隔小于冷却时间），验证仅推送第 1 次。

**Acceptance Scenarios**:

1. **Given** 上证指数涨超阈值异动冷却时间设为 30 分钟，**When** 10 分钟内连续 3 次触发同一异动，**Then** 仅第 1 次推送告警，后续 2 次被跳过并记录去重日志。
2. **Given** 两次异动事件间隔超过冷却时间，**When** 再次触发同类型异动，**Then** 正常推送新的告警消息。
3. **Given** 同一指数在不同条件下触发了不同类型异动（如涨超阈值 + 放量），**When** 冷却时间内的不同事件类型，**Then** 不同类型独立判断，均可正常推送。

---

### User Story 3 - 推送状态追溯 (Priority: P3)

作为运维人员，我希望查看近期告警推送的历史记录和推送状态（成功/失败/去重跳过），以便排查推送问题或统计告警频率。

**Why this priority**: 推送追溯是运营和排障的辅助能力，在基础推送功能稳定后才有实际价值。

**Independent Test**: 系统运行一段时间产生多条件告警后，请求推送历史接口，验证返回记录包含推送时间和状态。

**Acceptance Scenarios**:

1. **Given** 系统在过去 1 小时内推送了 5 条告警（3 成功、1 失败、1 去重），**When** 用户请求推送历史记录，**Then** 返回 5 条记录，每条包含事件 ID、状态和时间。
2. **Given** 尚无任何推送记录，**When** 用户请求推送历史，**Then** 返回空列表，状态码正常。
3. **Given** 推送历史缓存达到上限（如 1000 条），**When** 新推送记录产生，**Then** 最早记录被淘汰，系统正常运行。

---

### Edge Cases

- Webhook URL 变更后是否需要重启服务才能生效？（假设：需重启，与现有配置加载机制一致）
- 推送消息体过大（如描述字段超长）时如何截断？
- 企业微信 Webhook 返回非 200 状态码（如 429 限流、500 服务端错误）时的处理策略？
- 企业微信机器人消息发送频率受限（每分钟 20 条）时如何处理突发大量告警？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在异动事件生成后自动触发 Webhook 推送流程（推送与检测引擎异步解耦）。
- **FR-002**: 系统 MUST 支持企业微信机器人 Webhook 格式（POST JSON），消息使用 Markdown 类型，固定包含字段：指数名称、异动类型（中文）、触发值、阈值、触发时间。
- **FR-003**: 系统 MUST 支持通过配置文件启用或禁用企业微信推送，配置项包括 Webhook URL。
- **FR-004**: 系统 MUST 具备告警冷却机制：同一指数 + 同一异动类型在冷却时间内仅推送一次，冷却时间可配置。
- **FR-005**: 系统 MUST 在推送返回 5xx 错误、429 限流或网络超时时进行有限次重试（默认 3 次），重试间隔可配置；4xx 客户端错误（400/401/404）直接标记失败不再重试，记录错误日志。
- **FR-006**: 系统 MUST 将推送结果（成功/失败/去重跳过/重试中）记录到内存中的推送历史，并提供查询接口。
- **FR-007**: 推送失败或 Webhook 不可达 MUST NOT 阻塞异动检测流程或影响 API 正常响应。
- **FR-008**: 系统 MUST 在未配置 Webhook 时完全跳过推送流程（不影响现有功能）。
- **FR-009**: 同一轮检测产生多条异动事件时，系统 MUST 合并为一条 Markdown 消息推送，用分隔线区分各条异动。

### Key Entities

- **Webhook 配置 (WebhookConfig)**: Webhook URL、启用状态、冷却时间、重试次数与间隔。
- **推送目标 (NotifyTarget)**: 由 WebhookConfig 创建，负责将异动事件格式化为企业微信 Markdown 消息并发送。
- **推送记录 (DeliveryRecord)**: 事件 ID、推送时间、状态（成功/失败/去重/重试中）、重试次数、错误信息（如有）。
- **冷却追踪 (CooldownTracker)**: 指数代码 + 异动类型的组合键、最近一次推送时间，用于判断是否需要冷却跳过。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 异动事件从生成到企业微信推送完成的端到端延迟不超过 5 秒（网络正常情况下）。
- **SC-002**: Webhook 推送成功率不低于 95%（在网络和第三方服务正常的前提下）。
- **SC-003**: 告警冷却机制在冷却时间内的重复告警抑制率达到 100%。
- **SC-004**: 推送过程不阻塞异动检测接口的响应时间。
- **SC-005**: 运维人员可在 5 分钟内完成企业微信 Webhook 配置（包括获取 URL、填入配置文件、重启服务）。

## Assumptions

- 企业微信使用群机器人 Webhook 方式接入，用户需自行在企业微信群中创建机器人并获取 Webhook URL。
- 企业微信机器人消息频率限制（每分钟 20 条）由平台侧控制，合并推送策略可减少调用次数缓解限流。
- 默认冷却时间为 30 分钟，可通过配置调整；设为 0 表示禁用冷却。
- 默认重试次数为 3 次，重试间隔为 30 秒；可通过配置调整。
- 推送历史在内存中最多保留 1000 条记录，超出后按先进先出淘汰。
- 企业微信 Markdown 消息不支持表格和列表，消息以标题 + 字段罗列方式呈现，多异动之间用 `---` 分隔线区分。
- Webhook 配置在服务启动时加载，运行期间不支持热更新（与现有配置机制一致）。
- 推送流程与异动检测引擎之间通过异步非阻塞方式集成。
- 微信（个人号/服务号）不支持群机器人 Webhook，因此仅实现企业微信渠道。
